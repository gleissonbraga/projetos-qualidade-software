## Nível de maturidade estimado

**MPS.BR Nível G – Parcialmente Gerenciado**

No decorrer do processo, mostra-se que existe práticas básicas, como controle de versão e organização de artefatos.  Porém não há definição formal dos processos, das métricas ou rastreabilidade.  Sem gestão de tarefas ou acompanhamento sistemático.

## Lacunas Identificadas 
| Lacuna | Impacto |
| :--- | :--- |
| Ausência de registro d bugs e problemas [cite: 13] | Equipe acaba esquecendo alguns erros e às vezes o mesmo bug aparece de novo porque não ficou documentado.  |
| Falta de métricas de qualidade [cite: 13] | Quantidade de bugs, tempo de entrega, fica difícil saber se o projeto está melhorado ou não.  |
| Sem padronização [cite: 13] | Testes feitos de forma aleatória de acordo a cada pessoa. Fazendo a qualidade das entregas oscilarem.  |
| Sem rastreabilidade entre requisitos e funcionalidades [cite: 13] | Não dá para saber com certeza se tudo que foi planejado realmente foi implementado.|
| Retrospectivas desestruturas [cite: 13] | Falta de reunião da equipe faz com que os mesmos problemas se repitam.  |

## Melhorias Propostas

| Melhoria | Benefício |
| :--- | :--- |
| Padronizar testes para as funcionalidades | Regras mais confiáveis e todos testam da mesma maneira. |
| Documentar bugs e tarefas com o Issues no GitHub | Ajuda no acompanhamento do que deve ser corrigido e os problemas não se repetem. |
| Relacionar requisitos com Issues e pull requests | Ajuda a diagnosticar de fato as funcionalidades que atendem aos requisitos, aumentando a organização. |
| Pequenas retrospectivas ao fim de cada etapa | O grupo consegue identificar o que deu certo e o que precisa ser melhorado. Assim fazendo o processo evoluir aos poucos. |
| Definição de métricas básicas. – Números de bugs por semana e tarefas concluídas. | Permite verificar a evolução do projeto e assim tomar decisões com base em dados. |

---

## Conclusão 

O processo da equipe está num nível gerenciado: já existe organização básica e o uso do GitHub, porém falta práticas mais alinhadas, como métricas, registro de bugs e padronização de testes. Com ajustes corretos, o grupo pode se desenvolver rápido para um processo mais maduro.
